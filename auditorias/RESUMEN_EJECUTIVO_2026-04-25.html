📊 Resumen General

✅ Lo que está bien

  • Base técnica sólida. La plataforma está bien construida. El código es ordenado, organizado y sigue buenas prácticas.
  • Muchas pruebas automáticas. Hay 1,889 pruebas que se ejecutan solos para detectar errores antes de que lleguen al usuario. 0 fallando.
  • Cada empresa ve solo sus datos. El sistema de aislamiento multi-empresa funciona correctamente.
  • Dashboard rápido. Usa caché para no recalcular todo cada vez que alguien abre la página.
  • Almacenamiento en la nube. Archivos guardados en Cloudflare R2 con encriptación.
  • Chatbot IA integrado. "Señor SAM" con Gemini funciona y persiste la conversación.
  • Pagos con Wompi. Integración funcional con el procesador de pagos colombiano.
  • Generación de PDFs y Excel. Reportes metrológicos completos y exportables.
  • Documentación mejorada. README y guía de contribución reescritos y actualizados.
  • Score subió de 8.3 a 8.5 respecto a la auditoría de marzo 2026.

⚠️ Lo que preocupa

  • Los emails de recordatorio están rotos. La plataforma envía correos de vencimiento semanales, pero todos dicen "N/A días" en vez de los días reales. Nadie recibe información útil.
  • Vulnerabilidad en pagos. Hay una falla técnica en cómo se validan los pagos de Wompi que un atacante avanzado podría explotar.
  • Si el servidor de caché se cae, el dashboard falla para todos. No hay plan B si Redis no responde.
  • El chatbot no tiene límite de uso. Cualquier usuario puede enviar miles de mensajes y generar costos altos de API.
  • Las pruebas automáticas en CI no son iguales a producción. Se prueban en Python 3.11 pero el servidor real usa Python 3.13.
  • Si el almacenamiento de archivos falla, los datos quedan en estado inconsistente. Se guarda en la base de datos pero el archivo no llega.
  • Scripts de seguridad web desactualizados. La protección contra ataques tipo XSS no está completamente activada (52 secciones sin protección).
  • Código antiguo sin usar. Archivos que ya nadie usa siguen existiendo y hacen el proyecto más difícil de mantener.

🚨 Hallazgos Detallados

● Crítico
Los emails de recordatorio de vencimiento no funcionan
Cada semana la plataforma envía correos avisando qué equipos están por vencer. Pero todos dicen "N/A días" porque el sistema intenta leer información que no existe. Los usuarios reciben el correo pero sin datos reales — lo que hace que el aviso sea inútil.
AltoImpacto en
usuarios
● Crítico
Vulnerabilidad técnica en la validación de pagos (Wompi)
Cuando Wompi confirma un pago, la plataforma verifica que el mensaje venga realmente de Wompi comparando una firma. El método de comparación usado permite a un atacante muy sofisticado adivinar la firma midiendo cuánto tiempo tarda el servidor en responder. Solución: cambiar una sola línea de código.
AltoRiesgo
financiero
▲ Alto
El dashboard falla para todos si el servidor de caché cae
El sistema usa Redis (servidor de caché) para que el dashboard cargue rápido. Si Redis tiene un problema, el dashboard devuelve un error a todos los usuarios al mismo tiempo, en vez de simplemente cargarlo más lento. Se necesita un plan de respaldo.
MedioDisponibilidad
▲ Alto
Archivos pueden perderse si el almacenamiento en la nube falla a la mitad
Al subir un archivo (certificado de calibración, logo de empresa, etc.), el sistema primero registra el archivo en la base de datos y luego lo sube a Cloudflare R2. Si la subida a R2 falla, el registro queda en la base de datos pero el archivo no existe en ningún lado. No hay reintento automático ni aviso.
MedioIntegridad
de datos
▲ Alto
El chatbot "Señor SAM" no tiene límite de mensajes
Cualquier usuario puede enviar mensajes sin límite. Cada mensaje consume créditos de la API de Google Gemini (que se pagan). Un usuario malintencionado o un script automatizado podría generar costos elevados en poco tiempo sin que nadie lo note.
MedioCosto
operativo
▲ Alto
Las pruebas automáticas en CI usan una versión de Python diferente a producción
El servidor de producción corre Python 3.13. Las pruebas automáticas que se ejecutan al subir código al repositorio usan Python 3.11. Un error que solo ocurra en 3.13 podría llegar a producción sin ser detectado. Además, no se instalan todas las dependencias necesarias para correr los tests.
MedioRiesgo en
deploy
▲ Alto
Consultas lentas al dashboard con muchos equipos
En la gráfica de estado de los equipos, el sistema hace una consulta separada a la base de datos por cada equipo. Con 100 equipos, eso son 100 consultas extras cuando podrían hacerse todas en una sola. A mayor número de equipos por empresa, más lento el dashboard.
MedioVelocidad
◆ Medio
Protección avanzada contra ataques web (CSP) no está activa
Existe un tipo de ataque llamado XSS donde alguien inyecta código malicioso en la página. Para prevenirlo hay un mecanismo llamado CSP (política de seguridad de contenido). La plataforma lo tiene configurado pero desactivado en 52 secciones porque el código de la página está mezclado con el código de estilo/scripts. Requiere refactorización de 51 pantallas.

Algunas claves de caché se guardan en Redis sin fecha de expiración. Con el tiempo, Redis acumula datos que nadie usa, consumiendo memoria del servidor innecesariamente.

El historial de acciones del administrador del sistema se guarda en caché con una clave genérica (sin identificar la empresa). Esto significa que todas las empresas podrían ver el historial de otra. Es un problema de privacidad menor pero real.

La configuración del caché mezcla opciones de dos librerías distintas que no son totalmente compatibles. Funciona por ahora, pero puede generar comportamientos inesperados en condiciones de estrés.

La norma ISO/IEC 17020:2012 exige gestionar quejas de clientes (sección 7.9), no conformidades (8.7) e imparcialidad (4.1). La plataforma no tiene estos módulos. Dependiendo del nivel de cumplimiento que se quiera certificar, esto puede ser un bloqueador.
